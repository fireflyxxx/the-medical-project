# 医生端界面重构方案

## 一、现状分析

### 当前界面存在的问题

通过分析 `doctor3.0.html` 和用例模型文档，发现当前医生端界面存在以下核心问题：

1. **功能层级混乱**
   - 侧边栏导航：诊断工作台、病例管理、影像上传、统计报表
   - 这些功能模块的划分不符合医生的实际工作流程
   - 缺少明确的"病例 → 检查 → 影像 → AI推理"层级关系

2. **与用例模型不匹配**
   - 用例模型中的核心流程：创建病例 → 创建检查 → 上传影像 → 发起推理 → 查看结果 → 评价/修改
   - 当前界面将"影像上传"独立为一级菜单，割裂了病例-检查-影像的层级关系
   - "诊断工作台"混合了患者列表和AI分析结果，职责不清

3. **数据层级关系不清晰**
   - 病例（Patient Case）→ 检查记录（Examination）→ 影像（Image）→ 推理任务（Inference Task）
   - 当前界面没有体现这种层级关系，用户无法直观理解数据结构

---

## 二、重构目标

### 核心原则

1. **以用例为驱动**：界面设计严格遵循用例模型中定义的医生工作流程
2. **层级清晰**：体现"病例 → 检查 → 影像 → 推理"的数据层级关系
3. **操作连贯**：减少页面跳转，在同一上下文中完成相关操作
4. **符合医生习惯**：按照临床工作流程组织功能

---

## 三、重构后的功能架构

### 3.1 一级导航（侧边栏）

```
├── 工作台（Dashboard）
│   └── 概览统计 + 待处理任务提醒
│
├── 病例管理（Case Management）
│   ├── 病例列表（查看病例数据 - 用例2）
│   ├── 创建病例（创建病例数据 - 用例1）
│   └── 病例详情（进入二级页面）
│       ├── 病例基础信息（可编辑 - 用例3）
│       ├── 检查记录列表
│       │   ├── 创建检查（创建检查记录 - 用例4）
│       │   ├── 删除检查（删除检查记录 - 用例5）
│       │   └── 检查详情（进入三级页面）
│       │       ├── 影像管理
│       │       │   ├── 上传影像（上传影像数据 - 用例7）
│       │       │   ├── 查看影像（查看影像数据 - 用例8）
│       │       │   └── 删除影像（删除影像数据 - 用例9）
│       │       ├── AI推理
│       │       │   ├── 发起推理（发起肺炎推理任务 - 用例10）
│       │       │   ├── 查看结果（查看AI预测结果图 - 用例11）
│       │       │   ├── 修改检测框（修改AI预测结果图检测框 - 用例12）
│       │       │   ├── 评价结果（评价推理结果 - 用例13）
│       │       │   └── 导出结果（导出结果 - 用例14）
│       │       └── 推理历史记录
│       └── 删除病例（删除病例数据 - 用例6）
│
└── 统计分析（Statistics）
    └── 数据报表和图表
```

### 3.2 功能层级说明

#### 第一层：工作台
- **目的**：快速概览和任务提醒
- **内容**：今日统计、待处理病例数、高风险预警、快捷入口

#### 第二层：病例管理（核心工作区）
- **病例列表页**
  - 显示所有病例（用例2：查看病例数据）
  - 支持筛选：病例编号、时间范围
  - 操作：新建病例、查看详情、删除病例

- **病例详情页**（点击某个病例后进入）
  - 上部：病例基础信息（可编辑，用例3）
  - 中部：检查记录列表
    - 每条检查记录显示：检查时间、检查类型、影像数量、推理状态
    - 操作：新建检查、删除检查、进入检查详情

- **检查详情页**（点击某个检查记录后进入）
  - 左侧：影像列表
    - 显示该检查下的所有影像缩略图
    - 操作：上传影像、查看影像、删除影像
  - 右侧：AI推理区域
    - 选择影像 → 配置模型参数 → 发起推理
    - 推理结果展示：原图 + 标注图
    - 操作：修改检测框、评价结果、导出结果
  - 底部：推理历史记录
    - 显示该检查下所有推理任务的历史记录

#### 第三层：统计分析
- 数据报表、图表可视化（保留原有功能）

## 五、关键改进点

### 5.1 层级关系清晰化

**改进前**：
- 侧边栏：诊断工作台、病例管理、影像上传、统计报表（平级）
- 用户需要在不同页面间跳转才能完成"创建病例 → 上传影像 → 推理"流程

**改进后**：
- 病例 → 检查 → 影像/推理（层级递进）
- 在同一上下文中完成所有操作，减少认知负担

### 5.2 操作流程连贯性

**改进前**：
- 在"病例管理"创建病例
- 跳转到"影像上传"上传影像
- 回到"诊断工作台"查看推理结果

**改进后**：
- 在病例详情页创建检查
- 在检查详情页上传影像
- 在同一页面发起推理、查看结果、评价、导出
- 完整的工作流在一个页面内完成

### 5.3 符合用例模型

| 用例编号 | 用例名称 | 对应界面位置 |
|---------|---------|-------------|

## 六、实施建议

### 6.1 重构优先级

**P0（高优先级）**：
1. 重构侧边栏导航结构
2. 实现病例列表页和病例详情页
3. 实现检查详情页（核心工作页面）

**P1（中优先级）**：
4. 完善影像管理功能
5. 完善AI推理功能
6. 实现推理历史记录

**P2（低优先级）**：
7. 优化工作台统计展示
8. 优化统计分析页面

### 6.2 技术实现要点

1. **路由设计**
   - `/cases` - 病例列表
   - `/cases/:caseId` - 病例详情
   - `/cases/:caseId/examinations/:examId` - 检查详情

2. **状态管理**
   - 当前病例上下文
   - 当前检查上下文
   - 面包屑导航状态

3. **数据加载策略**
   - 病例列表：分页加载
   - 病例详情：懒加载检查记录
   - 检查详情：懒加载影像和推理记录

### 6.3 用户体验优化

1. **面包屑导航**：清晰显示当前位置和层级关系
2. **快捷操作**：在列表页提供快捷操作按钮
3. **状态提示**：明确显示推理任务状态（待处理/执行中/成功/失败）
4. **二次确认**：删除操作需要二次确认（符合用例要求）
5. **错误处理**：按照用例中的异常事件流处理各种错误情况

---

## 七、总结

本重构方案的核心思想是：

1. **以数据层级为骨架**：病例 → 检查 → 影像 → 推理
2. **以用例流程为血肉**：每个用例在界面中都有明确的入口和操作路径
3. **以医生工作流为灵魂**：界面组织符合临床实际工作习惯

通过这次重构，医生端界面将从"功能堆砌"转变为"流程驱动"，大幅提升可用性和工作效率。
