# 产品需求文档 (PRD)
## 医疗影像AI分析平台

---

## 1. 产品概述

### 1.1 产品定位
基于AI驱动的医疗影像分析平台，专注于辅助医生进行肺炎等疾病的影像诊断。系统通过本地AI模型集成，实现从影像上传、AI推理到结果导出的完整闭环，提升诊断效率和准确性。

### 1.2 目标用户
- **超声科医生**：平台主要使用者，负责上传影像、发起AI检测、查看和评价结果
- **科研人员**：查看医生提交的案例与AI识别结果，根据反馈调整AI模型
- **系统管理员**：负责用户管理、系统配置及运维监控

### 1.3 核心价值
- 提升医生诊断效率，减少漏诊误诊
- 建立AI辅助诊断的人工闭环反馈机制
- 保障医疗数据安全与合规性
- 为科研人员提供真实临床数据支持

## 2. 功能需求

### 2.1 账号与权限管理 (FR-01)

#### 2.1.1 用户登录
**功能描述**：用户通过用户名和密码登录系统，获取访问令牌
- 支持账号密码验证
- 登录失败3次后锁定账号15分钟
- 记录登录日志（时间、IP、设备）

#### 2.1.2 角色权限控制 (RBAC)
**权限矩阵**：

| 功能模块 | 医生 | 科研人员 | 管理员 |
|---------|------|---------|--------|
| 创建/编辑病例 | ✓ | ✗ | ✗ |
| 上传/删除影像 | ✓ | ✗ | ✗ |
| 发起AI推理 | ✓ | ✗ | ✗ |
| 评价结果 | ✓ | ✗ | ✗ |
| 查看所有数据 | ✗ | ✓ | ✓ |
| 用户管理 | ✗ | ✗ | ✓ |
| 系统日志 | ✗ | ✗ | ✓ |

#### 2.1.3 账号管理（管理员）
- 创建账号：填写用户名、初始密码、角色
- 禁用账号：停用违规或离职用户
- 密码策略：至少8位，包含字母+数字

---

### 2.2 病例数据管理 (FR-02)

#### 2.2.1 创建病例
**必填字段**：
- 患者姓名（自动脱敏：张** → 张*）
- 身份证号（自动脱敏：320123...1234 → 320***）
- 检查时间
- 检查类型（胸部X光、CT等）

**校验规则**：
- 前端：必填项检查、格式验证
- 后端：二级校验 + 脱敏处理

#### 2.2.2 病例检索
**检索条件**：
- 病例编号（精确匹配）
- 时间范围（起止日期）
- 检查类型（下拉选择）

**展示内容**：
- 列表：病例编号、脱敏姓名、检查时间
- 详情：基础信息 + 关联检查记录

#### 2.2.3 编辑与删除
- 编辑：支持修改病例基础信息和检查描述
- 删除：二次确认，级联删除关联影像和推理记录
- 异常拦截：存在执行中的推理任务时禁止删除

---

### 2.3 影像数据管理

#### 2.3.1 上传影像
**支持格式**：PNG、JPG、DICOM
**文件限制**：单文件 ≤ 50MB
**校验流程**：
1. 前端预校验：格式 + 大小
2. 后端合规检查：内容识别（防止非医学影像）
3. OSS存储：返回文件路径
4. 数据库记录：路径、格式、上传时间

**进度反馈**：
- 上传中：显示进度条（0% → 100%）
- 成功：展示缩略图
- 失败：明确错误提示

#### 2.3.2 查看影像
- 支持在线预览（放大/缩小/拖拽）
- DICOM格式支持窗宽窗位调整
- 加载失败显示占位图

#### 2.3.3 删除影像
- 二次确认弹窗
- 物理删除OSS文件 + 逻辑删除数据库记录
- 事务保证一致性

---

### 2.4 AI推理任务 (FR-03, FR-04)

#### 2.4.1 发起推理
**操作流程**：
1. 选择影像（支持多选）
2. 选择AI模型（肺炎检测模型）
3. 配置参数（置信度阈值：0.5-0.9）
4. 点击"开始AI辅助检测"

**异步处理**：
- 前端：提交后立即返回 HTTP 202，显示"推理已提交"
- 后端：任务进入消息队列，状态初始化为 PENDING
- AI模块：异步执行推理，完成后更新状态为 SUCCESS/FAILED

#### 2.4.2 任务状态管理
**状态流转**：
```
PENDING（待处理） → IN_PROGRESS（执行中） → SUCCESS（成功）
                                        ↘ FAILED（失败）
```

**存储内容**：
- 任务ID、状态、开始时间、结束时间
- 推理耗时（毫秒）
- 推理结果（JSON格式）
- 错误信息（失败时）

#### 2.4.3 结果查询
- 轮询方式：前端每3秒查询一次任务状态
- WebSocket推送（可选）：实时推送状态变更

---

### 2.5 结果可视化与交互 (FR-05)

#### 2.5.1 结果展示
**展示内容**：
- 原始影像 + AI检测框叠加
- 分类结果：Normal / Pneumonia
- 置信度：0.00 - 1.00（百分比显示）
- 检测框坐标：(x, y, width, height)

**交互功能**：
- 阈值过滤：滑动条调整置信度阈值，动态过滤检测框
- 检测框高亮：鼠标悬停显示详细信息

#### 2.5.2 人工修正
**编辑模式**：
- 调整检测框大小和位置
- 删除误报框
- 新增漏报区域
- 保存后更新数据库坐标记录

**对比展示**：
- 左侧：AI原始预测框
- 右侧：医生修正框（Ground Truth）

#### 2.5.3 评价反馈
**评价选项**：
- 准确
- 存在误报
- 存在漏报

**补充说明**：文本框（选填，最多200字）

## 3. 非功能需求

### 3.1 性能要求
- 页面加载时间 ≤ 2秒
- AI推理响应时间 ≤ 5秒（单张影像）

### 3.2 安全要求
- 数据传输：HTTPS加密

- Token有效期：2小时，支持刷新

### 3.3 可用性要求
- 系统可用性 ≥ 99.5%
- 数据备份：每日自动备份
- 异常降级：AI服务异常时提示用户，不影响其他功能

### 3.4 兼容性要求
- 浏览器：Chrome 90+、Edge 90+、Firefox 88+
- 分辨率：1920x1080及以上
- 操作系统：Windows 10+、macOS 11+

## 4. 数据模型

### 4.1 核心实体

#### Case（病例）
```
- case_id: 主键
- patient_name_masked: 脱敏姓名
- id_number_masked: 脱敏身份证
- created_at: 创建时间
- updated_at: 更新时间
- is_deleted: 逻辑删除标记
```

#### Exam（检查记录）
```
- exam_id: 主键
- case_id: 外键
- exam_type: 检查类型
- exam_date: 检查日期
- description: 检查描述
- status: 状态（NORMAL/IN_PROGRESS）
```

#### Image（影像）
```
- image_id: 主键
- exam_id: 外键
- file_path: OSS路径
- file_format: 格式（PNG/JPG/DICOM）
- file_size: 文件大小
- uploaded_at: 上传时间
```

#### InferenceTask（推理任务）
```
- task_id: 主键
- image_id: 外键
- model_name: 模型名称
- status: 状态（PENDING/SUCCESS/FAILED）
- result: 推理结果（JSON）
- inference_time_ms: 推理耗时
- created_at: 创建时间
- completed_at: 完成时间
```

#### Evaluation（评价）
```
- evaluation_id: 主键
- task_id: 外键
- rating: 评价（ACCURATE/FALSE_POSITIVE/FALSE_NEGATIVE）
- comment: 补充说明
- created_by: 评价人
- created_at: 评价时间
```

## 5. 接口设计

### 5.1 RESTful API规范

#### 病例管理
```
POST   /api/v1/cases              # 创建病例
GET    /api/v1/cases              # 检索病例列表
GET    /api/v1/cases/{case_id}    # 获取病例详情
PUT    /api/v1/cases/{case_id}    # 更新病例
DELETE /api/v1/cases/{case_id}    # 删除病例
```

#### 检查记录
```
POST   /api/v1/cases/{case_id}/exams              # 创建检查记录
DELETE /api/v1/cases/{case_id}/exams/{exam_id}    # 删除检查记录
```

#### 影像管理
```
POST   /api/v1/cases/{case_id}/exams/{exam_id}/images        # 上传影像
GET    /api/v1/cases/{case_id}/exams/{exam_id}/images        # 获取影像列表
DELETE /api/v1/cases/{case_id}/exams/{exam_id}/images/{id}   # 删除影像
```

#### AI推理
```
POST   /api/v1/inference/pneumonia           # 发起推理任务
GET    /api/v1/inference/tasks/{task_id}     # 查询任务状态
PUT    /api/v1/inference/tasks/{task_id}     # 更新推理结果（修正检测框）
```

#### 评价管理
```
POST   /api/v1/evaluations                   # 提交评价
DELETE /api/v1/evaluations/{evaluation_id}   # 删除评价
```

### 5.2 状态码规范
- 200 OK：请求成功
- 201 Created：创建成功
- 202 Accepted：异步任务已接受
- 400 Bad Request：请求参数错误
- 401 Unauthorized：未授权
- 403 Forbidden：权限不足
- 404 Not Found：资源不存在
- 409 Conflict：资源冲突（如任务执行中）
- 422 Unprocessable Entity：语义错误（如内容不合规）
- 500 Internal Server Error：服务器错误

## 6. 用户故事

### 6.1 医生视角
**故事1**：作为医生，我希望快速上传患者的胸部X光片并获得AI辅助诊断结果，以便提高诊断效率。

**验收标准**：
- 上传影像后3秒内收到"推理已提交"提示
- 推理完成后自动刷新结果页面
- 结果页面清晰展示检测框和置信度

**故事2**：作为医生，我希望能修正AI的误报和漏报，以便为科研人员提供准确的Ground Truth。

**验收标准**：
- 进入编辑模式后可拖拽调整检测框
- 保存后系统记录修正前后的对比数据
- 修正操作被记录到审计日志

### 6.2 科研人员视角
**故事3**：作为科研人员，我希望查看所有医生的评价数据，以便评估模型性能并进行优化。

**验收标准**：
- 可按模型版本筛选评价数据
- 可导出评价统计报告（准确率、误报率、漏报率）
- 可对比AI预测框与医生修正框

### 6.3 管理员视角
**故事4**：作为管理员，我希望能追溯所有用户的敏感操作，以便满足医疗合规审计要求。

**验收标准**：
- 审计日志记录所有关键操作
- 可按用户、操作类型、时间范围检索
- 日志不可篡改，支持导出

## 8. 风险与应对

### 8.1 技术风险
**风险**：AI模型推理耗时过长，影响用户体验
**应对**：采用异步队列 + 轮询/WebSocket机制，避免前端长时间等待

### 8.2 合规风险
**风险**：敏感数据泄露导致合规问题
**应对**：全流程脱敏处理 + RBAC权限控制 + 操作审计

### 8.3 性能风险
**风险**：高并发场景下系统响应缓慢
**应对**：引入Redis缓存 + 数据库索引优化 + CDN加速
