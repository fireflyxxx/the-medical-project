用例描述：
1. 用例名：创建病例数据
用例简述：医生在系统中录入患者基础信息与检查任务信息，为上传影像做准备。
- 参与者：医生
- 前置条件：医生已成功登录系统并进入工作台。
- 后置条件：系统成功生成并保存新的病例记录和检查记录。
- 基本事件流：
1. 医生选择“新建病例”。
2. 系统展示待填表单
3. 医生在表单中填写患者基础信息和检查信息后保存。
4. 系统对提交的敏感字段信息执行基础脱敏处理，并在数据库中生成病例记录，提示创建成功。
- 异常事件流：
3.a. 必填项为空：系统提示“请完善必填信息”，阻止提交。
3.b. 违反格式限制：系统提示“输入格式错误”，阻止提交。

2. 用例名：查看病例数据
- 用例简述：医生浏览和检索系统中已存在的病例及检查记录。
- 参与者：医生
- 前置条件：医生已成功登录系统并进入工作台。
- 后置条件：无特定系统状态改变。
- 基本事件流：
1. 医生可通过输入病例编号或选择时间范围进行检索筛选。
2. 系统展示筛选后的病例。
3. 医生选择列表中某一条病例。
4. 系统展示该病例脱敏后的基础信息与关联的检查信息详情。
- 异常事件流：
无
  
3. 用例名：更改病例数据
- 用例简述：医生修改已有病例的基础信息或检查描述。
- 参与者：医生
- 前置条件：医生已成功登录系统，正在查看某个病例的详情。
- 后置条件：系统保存更新后的病例/检查信息。
- 基本事件流：
1. 医生在病例详情页选择“编辑”操作。
2. 系统将病例信息变为可编辑。
3. 医生修改所需的病例字段后保存。
4. 系统校验信息合法性后，更新数据库记录并提示“修改成功”。
- 异常事件流：
3.a. 必填项为空：系统提示“请完善必填信息”，阻止提交。
3.b. 修改后违反格式限制：系统提示“输入格式错误”，阻止提交。

4. 用例名：创建检查记录
用例简述：医生在系统中录入患者检查记录，为上传影像做准备。
- 参与者：医生
- 前置条件：医生已成功登录系统并进入某个病例详情。
- 后置条件：系统成功生成并保存新的病例记录和检查记录。
- 基本事件流：
1. 医生选择“新建检查”。
2. 系统展示待填表单
3. 医生在表单中填写所需信息后保存。
4. 系统对提交的敏感字段信息执行基础脱敏处理，并在数据库中更新病例记录，提示创建成功。
- 异常事件流：
3.a. 必填项为空：系统提示“请完善必填信息”，阻止提交。
3.b. 违反格式限制：系统提示“输入格式错误”，阻止提交。

5. 用例名：删除检查记录
- 用例简述：医生从系统中移除建错的或不再需要的检查记录及其关联的所有数据。
- 参与者：医生
- 前置条件：医生已成功登录系统并进入某个病例详情。
- 后置条件：选定的检查记录及其挂载的影像和推理记录被删除。
- 基本事件流：
1.  医生选中目标检查记录，选择“删除”。
2. 系统弹出二次确认框，警示将同步删除关联影像与任务。
3. 医生确认。
4. 系统清理相关数据，更新列表并提示“删除成功”。
- 异常事件流：
1.a. 该检查记录下存在“执行中”的推理任务：系统拒绝删除请求，提示“存在正在执行的AI推理任务，请稍后再试”。
6. 用例名：删除病例数据
- 用例简述：医生从系统中移除建错的或不再需要的病例及其关联的所有数据。
- 参与者：医生
- 前置条件：医生已成功登录系统并进入工作台页。
- 后置条件：选定的病例及其挂载的影像和推理记录被删除。
- 基本事件流：
1.  医生选中目标病例，选择“删除”。
2. 系统弹出二次确认，警示将同步删除关联影像与任务。
3. 医生确认。
4. 系统清理相关数据，更新列表并提示“删除成功”。
- 异常事件流：
1.a. 该病例下存在“执行中”的推理任务：系统拒绝删除请求，提示“存在正在执行的AI推理任务，请稍后再试”。
  
7. 用例名：上传影像数据
- 用例简述：医生向已选择的病例的检查记录中上传胸部X光图像文件。
- 参与者：医生
- 前置条件：医生已登录，并定位到具体的病例记录的下。
- 后置条件：影像持久化存储并与当前病例绑定。
- 基本事件流：
1. 医生在病例详情页某个检查记录详情选择“上传影像”。
2. 医生在本地选择需要上传的医学图像文件（支持 PNG、JPG 或 DICOM 格式）。
3. 系统展示上传进度条。
4. 上传完成后，系统记录文件路径、格式及上传时间，更新并在病例数据和检查记录中展示缩略图或文件名。
- 异常事件流：
2.a. 格式不符：医生上传了非指定格式的文件，系统前端拦截并提示“仅支持PNG/JPG/DICOM格式”。
2.a. 超过大小限制：系统提示“文件过大，请压缩或更换文件”。
2.a. 图片不合规：系统提示“图片内容异常，请检查或更换图片”。
  
8. 用例名：查看影像数据
- 用例简述：医生在线浏览已上传的影像原图。
- 参与者：医生
- 前置条件：医生已登录，当前病例下存在上传成功的影像文件。
- 后置条件：无。
- 基本事件流：
1. 医生在检查记录中选择指定的影像文件。
2. 系统展示影像文件。（要支持基础的放大/缩小查看。）
- 异常事件流：
2.a. 影像加载失败：前端展示加载失败占位图，并提示“影像文件加载失败”。
  
9. 用例名：删除影像数据
- 用例简述：医生删除上传错误或质量不达标的影像文件。
- 参与者：医生
- 前置条件：医生已登录，待删除影像未在处理流程中。
- 后置条件：指定的影像文件及其元数据记录被删除。
- 基本事件流：
1. 医生在某个检查记录中选择删除某个影像。
2. 系统弹出二次确认，警示将永久删除。
3. 医生确认。
4. 系统清理相关数据，更新列表并提示“删除成功”。
- 异常事件流：
4.a. 影像删除失败：系统提示删除失败。
  
10. 用例名：发起肺炎推理任务
- 用例简述：医生针对某张胸部X光影像让AI进行肺炎自动检测。
- 参与者：医生
- 前置条件：医生已登录，且已选择待分析的影像文件。
- 后置条件：系统生成状态为“待处理”或“成功”的推理任务记录。
- 基本事件流：
1. 医生选中一张或多张待测胸片，选择要调用的本地AI模型与参数配置。
2. 医生选择“AI辅助肺炎检测”。
3. 系统向模型传递影像数据。
4. 系统后台记录任务状态，返回“推理已提交，请稍候”。
5. 系统预处理图像并完成模型计算，保存最终输出。
- 异常事件流：
4.a. AI服务异常或崩溃：系统捕获异常，将任务状态标为“失败”，提示明确错误码及反馈。
- 特殊需求：
  - 推理请求后要保存任务状态、耗时、推理结果
  - 任务有状态流转（待处理/成功/失败）
  - 历史可查询
  
11. 用例名：查看AI预测结果图
- 用例简述：医生在前端可视化界面查看AI返回的推理分析结论与病灶标注图。
- 参与者：医生
- 前置条件：相应的推理任务已成功执行完成。
- 后置条件：无。
- 基本事件流：
1. 医生打开已完成推理的任务结果页。
2. 系统展示原图，并叠加展示检测框或掩码。界面同步呈现AI判定的类别、概率置信度以及坐标信息。
- 异常事件流：
2.a. 结果未准备好：若使用异步队列处理，医生提前点击查看时，系统提示“正在推理中，请刷新后重试”。
  
12. 用例名：修改AI预测结果图检测框
- 用例简述：医生针对AI辅助生成的病灶检测框进行人工干预修正，调整至准确位置。
- 参与者：医生
- 前置条件：医生正在查看包含检测框的AI预测结果图。
- 后置条件：系统保存修正后的坐标，实现人工闭环。
- 基本事件流：
1. 医生在界面上激活“编辑模式”。
2. 医生调整现有检测框的大小与位置，或删除误报框、框选漏报区域。
3. 医生“保存”。
4. 系统在数据库中更新对应的检测狂坐标记录，并刷新生成新的可视化标注图。
- 异常事件流：
2.a. 绘制的坐标超出原图尺寸：系统自动吸附边缘或弹出提示“检测框范围不合法”。
  
13. 用例名：评价推理结果
- 用例简述：医生对AI返回的诊断结论进行主观反馈，供后续科研人员进行多模型对比和人工闭环修正。
- 参与者：医生
- 前置条件：推理任务已完成。
- 后置条件：系统保存对该次推理的反馈评价数据。
- 基本事件流：
1. 医生在查看AI结果后，点击界面上的“评价”按钮。
2. 医生选择满意度评价（如“准确”、“误报”、“漏报”）并可选填写补充文本意见。
3. 医生提交表单。
4. 系统记录评价信息，更新反馈状态。
- 异常事件流：
4.a. 网络断开：提交时超时，系统提示“网络异常，评价未保存”。
  
12. 用例名：删除评价
- 用例简述：医生撤销自己以往提交给该病例结果的评价。
- 参与者：医生
- 前置条件：医生针对当前推理结果曾成功提交过评价记录。
- 后置条件：相关评价数据被清空。
- 基本事件流：
1. 医生在结果界面定位到自己已提交的评价信息区域。
2. 点击“删除评价”按钮。
3. 系统清除评价字段。
- 异常事件流：无。
  
13. 用例名：导出结果
- 用例简述：医生将AI最终产生的结构化结果和标注过的诊断图像打包下载，用于写诊断报告。
- 参与者：医生
- 前置条件：推理任务已完成，且有完整的推断结果。
- 后置条件：触发本地文件下载，操作被记入审计日志。
- 基本事件流：
1. 医生在结果交付界面点击“导出结果”。
2. 系统将结构化数据（分类、置信度、坐标等）打包为 JSON 格式文件，并附带生成的标注图。
3. 浏览器开始下载包含 JSON 和图片文件的压缩包。
- 异常事件流：
2.a. 未找到结果资产：后端缺失生成结果图片的路径，系统弹窗提示“导出失败，文件缺失，请重新触发推理”。

14. 用例名：查看所有病例数据
- 用例简述：科研人员跨越单个医生的权限范围，全局浏览和检索系统内所有已建档的脱敏病例及检查信息，以获取整体数据分布与样本概况。
- 参与者：科研人员
- 前置条件：科研人员已使用专有账号成功登录系统，系统内已存在由医生录入的病例记录。
- 后置条件：无特定系统状态改变。
- 基本事件流：
1. 科研人员点击进入工作台。
2. 系统列表展示所有脱敏处理后的病例记录和检查记录，如病例编号、检查时间、模态等。
3. 科研人员输入时间范围或病例编号进行筛选检索。
4. 选中特定病例后，系统展示该病例的基础信息与包含的所有检查描述。
- 异常事件流：
2.a. 数据访问越权拦截：如果系统RBAC权限配置异常，尝试加载含有敏感信息（PII）的未脱敏字段，系统底层将触发合规拦截，提示“无权查看敏感明文数据”。
  
15. 用例名：查看所有影像数据和AI推理详细结果
- 用例简述：科研人员浏览系统内汇聚的所有医学影像资产，用于构建训练集、测试集评估或算法前期的图像质量检查。
- 参与者：科研人员
- 前置条件：科研人员已登录，且系统存储中存在上传成功的医学影像。
- 后置条件：无。
- 基本事件流：
1. 科研人员进入全局“影像资产池”页面。
2. 系统展示所有影像数据的缩略图及基础元数据（上传时间、格式等）。
3. 科研人员选择任意一张影像。
4. 系统展示高清原图，科研人员可进行缩放、拖拽观察肺部细节特征。
- 异常事件流：
4.a. 文件失效：若底层文件被误删或存储服务断开，前端图片加载失败并显示占位图，系统提示“影像文件读取失败（文件不存在或路径错误）”。
  
16. 用例名：查看AI推理详细结果
- 用例简述：科研人员深度查看AI模型输出的各项底层指标，包括任务耗时、置信度、检测框坐标等，用于评估基线模型表现或进行多模型对比。
- 参与者：科研人员
- 前置条件：科研人员已登录，系统中存在已执行完成（状态为“成功”或“失败”）的AI推理任务。
- 后置条件：无。
- 基本事件流：
1. 科研人员进入“AI分析域”模块。
2. 系统展示所有历史推理任务，包括任务状态、耗时、所用模型版本等信息。
3. 科研人员选择某次成功的推理任务。
4. 系统展开详细结构化结果（InferenceResult），展示：分类标签（Normal/Pneumonia）、具体的0~1浮点数概率置信度。
5. 系统在可视化界面呈现原图并叠加AI预测的检测框，同时在侧边栏展示具体的坐标数据（x, y, width, height）。
- 异常事件流：
3.a. 推理任务失败：若科研人员点击状态为“失败”的任务，系统不展示图像结果，而是展示关键错误追踪信息（Request ID、错误码及系统运行日志截取），辅助科研人员排查模型崩溃原因。
  
17. 用例名：查看医生评价信息
- 用例简述：科研人员查看临床医生对AI推理结果做出的满意度评价及人工修正的标注数据，从而获取“人工闭环”反馈，为后续模型调优提供真实参考。
- 参与者：科研人员
- 前置条件：科研人员已登录，且对应的推理结果已经有医生提交了评价内容或修改过Bbox坐标。
- 后置条件：无。
- 基本事件流：
1. 科研人员在查看某条推理详细结果时，切换至“评价”标签页。
2. 系统展示该条影像对应的主治医生评价数据（如：“准确”、“存在误报/漏报”等标签）。
3. 系统展示医生填写的补充文本描述。
4. 如果医生对检测框进行过人工修改，系统将同屏对比显示“AI原始预测框”与“医生修正框”（即Ground Truth），并展示位置差异。
- 异常事件流：
3.a. 该数据尚未被评价：系统在此模块显示空白状态提示“临床医生尚未对该结果进行评价”。

18. 用例名：创建用户账号
- 用例简述：管理员在系统中为新加入的医生、科研人员或其他管理员分配登录账号
- 参与者：管理员
- 前置条件：管理员成功登录系统。
- 后置条件：生成新的账号记录，新用户可使用该凭证登录系统。
- 基本事件流：
1. 管理员导航至“用户管理”模块，点击“新建账号”按钮。
2. 管理员在表单中填写新用户的基本信息（如用户名、初始密码、姓名/工号等）。
3. 管理员为该账号指定系统角色（选择“医生”、“科研人员”或“管理员”）。
4. 管理员确认无误后点击“保存提交”。
5. 系统校验信息合法性后写入数据库，提示“账号创建成功”。
- 异常事件流：
2.a. 账号重名冲突：若输入的用户名在系统中已存在，系统前端拦截提交，并提示“该用户名已被占用，请更换”。
2.a. 密码强度不足：输入的初始密码不符合系统安全策略（如缺少字母数字组合），系统提示“密码复杂度过低”。
  
19. 用例名：禁用用户账号
- 用例简述：管理员对违规操作、离职或不再需要平台访问权限的用户账号进行停用处理，确保系统数据安全与合规。
- 参与者：系统管理员
- 前置条件：管理员已成功登录系统，目标账号当前处于“启用”状态。
- 后置条件：目标账号的状态字段（账号状态）变更为“禁用”，该账号无法再登录系统或调用任何业务鉴权接口。
- 基本事件流：
1. 管理员在“用户管理”列表中通过搜索或翻页定位到目标账号。
2. 管理员选择该账号操作栏中的“禁用”按钮。
3. 系统弹出二次安全确认弹窗，提示“禁用后该用户将无法登录，是否继续？”。
4. 管理员点击“确认”。
5. 系统更新数据库中的用户状态，强制使得该用户的当前Token失效（若正在线），并提示“账号已禁用”。
- 异常事件流：
2.a. 尝试禁用自身：管理员尝试禁用当前正在使用的账号时，系统拦截该操作，提示“无法禁用当前登录的管理员账号”。
  
20. 用例名：查看系统运行日志
- 用例简述：管理员查看后端系统底层（如API服务、AI推理服务）的运行状态、错误追踪等日志，以排查故障并保障系统的可靠性（Reliability）。
- 参与者：管理员
- 前置条件：管理员已成功登录系统并进入运维管理工作台。
- 后置条件：无特定数据状态改变。
- 基本事件流：
1. 管理员进入“系统运行日志”界面。
2. 系统默认展示近期的后端运行日志（Info/Warning/Error级别）。
3. 管理员可通过输入 Request ID、错误码或选择特定时间范围进行精准检索。
4. 管理员选择某条异常日志（如AI推理服务崩溃、Graceful Degradation触发记录），查看详细的错误堆栈信息，用于后续技术排查。
- 异常事件流：
4.a. 日志数据量过大导致查询超时：若管理员选择的时间跨度过大，系统查询过慢，界面提示“查询超时，请缩小时间范围后重试”。
  
21. 用例名：审计敏感操作
- 用例简述：管理员查看和追溯系统内全体用户的关键行为操作记录，以满足安全合规要求，实现“操作可追溯”的完整闭环。
- 参与者：系统管理员
- 前置条件：管理员已成功登录系统；系统审计组件已自动记录了相关用户的行为。
- 后置条件：无。
- 基本事件流：
1. 管理员导航至“操作审计/审计日志”模块。
2. 管理员在筛选条件中选择特定操作人（如某位医生）、操作类型（如登录、上传影像、触发推理、导出结果等）或时间段。
3. 列表按时间倒序展示符合条件的审计日志，清晰列出：操作人、操作类型、发生时间以及操作的对象ID（如病例编号、任务ID）。
- 异常事件流：
3.a. 无相关记录：管理员输入的检索条件过于严苛或目标用户无任何操作，系统列表显示空白，提示“暂无相关审计操作记录”。

22. 用例名：登录账号
  简述：用户输入预分配的用户名与密码，通过鉴权校验获取系统访问令牌（Token），进入对应角色的工作台。
23. 用例名：退出账号
  简述：用户完成工作后，主动点击“退出登录”，系统销毁当前的会话凭证，退回至登录页面，防止他人越权操作
