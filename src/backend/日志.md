**1. 接口规划（POST /api/v1/log/get）**

1. 接口定义
   POST /api/v1/log/get
   用途：分页查询审计日志（支持条件筛选）。
   权限：建议仅 admin 可用。
2. 请求体

```
{
  "operator": "doctorKT",
  "operation_type": "login",
  "target_id": "IMG1234567890",
  "target_type": "image",
  "operation_status": 1,
  "start_time": "202603010000",
  "end_time": "202603312359",
  "page_no": 1,
  "page_size": 20
}
```

字段说明：

- operator 可选，操作人
- operation_type 可选，操作类型
- target_id 可选，目标对象ID
- target_type 可选，目标对象类型
- operation_status 可选，1 成功，0 失败
- start_time/end_time 可选，格式 yyyyMMddHHmm
- page_no/page_size 必填，分页参数

1. 响应体

```
{
  "auditlog": [
    {
      "operator": "doctorKT",
      "operation_type": "login",
      "operation_time": "202603091106",
      "target_id": "doctorKT",
      "target_type": "user",
      "operation_status": 1,
      "error_msg": null
    }
  ],
  "total": 125,
  "page_no": 1,
  "page_size": 20
}
```

1. 规则
   - operation_time 对外固定输出 12 位：yyyyMMddHHmm
   - 默认按 operation_time desc 排序
   - error_msg 仅失败时有值，成功时 null

**2. 哪些地方要加日志（“查看/导出”除外）**

**2.1 必须记录日志的接口（写操作）**

1. 账号模块
   - POST /api/v1/account/create -> operation_type=create_user
   - POST /api/v1/account/login -> operation_type=login
   - POST /api/v1/account/{user_id}/ban -> operation_type=ban_user
2. 病例模块
   - POST /api/v1/cases/create -> create_case
   - POST /api/v1/cases/{case_id}/update -> update_case
   - POST /api/v1/cases/{case_id}/delete -> delete_case
   - POST /api/v1/cases/dev/clear_all -> clear_all_case
   - POST /api/v1/cases/{case_id}/study/create -> create_study
   - POST /api/v1/cases/{case_id}/study/{study_id}/update -> update_study
   - POST /api/v1/cases/{case_id}/study/{study_id}/delete -> delete_study
   - POST /api/v1/cases/{case_id}/study/{study_id}/upload_image -> upload_image
3. 影像/推理/评论模块
   - POST /api/v1/image/delete/{image_id} -> delete_image
   - POST /api/v1/image/infer/{image_id} -> infer_image
   - POST /api/v1/ai/infer -> infer_image（同义入口）
   - POST /api/v1/infer_result/{result_id}/comment/create -> create_comment
   - POST /api/v1/infer_result/{result_id}/comment/{comment_id}/delete -> delete_comment

**2.2 明确不记录（查看类）**

1. GET /api/v1/account/get
2. GET /api/v1/cases/get
3. GET /api/v1/image/get
4. GET /api/v1/image/get/{image_id}
5. GET /api/v1/infer_result/get
6. POST /api/v1/infer_result/get/{result_id}（语义是查询）
7. POST /api/v1/log/get（日志查询本身）

**2.3 记录口径（必须统一）**

1. 每个写操作都记录 成功 和 失败
2. 成功：operation_status=1, error_msg=null
3. 失败：operation_status=0, error_msg=异常信息
4. operator 取 token 中 username（登录失败场景取请求里的用户名）
5. target_type/target_id 按操作对象填（user/case/study/image/result/comment/system）

**3. 日志数据表**

```
CREATE TABLE IF NOT EXISTS audit_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    operator VARCHAR(64) NOT NULL COMMENT '操作人(username)',
    operation_type VARCHAR(64) NOT NULL COMMENT '操作类型',
    operation_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '操作时间',
    target_id VARCHAR(64) NOT NULL COMMENT '操作对象ID',
    target_type VARCHAR(32) NOT NULL COMMENT '操作对象类型',
    operation_status TINYINT NOT NULL COMMENT '1成功, 0失败',
    error_msg VARCHAR(512) NULL COMMENT '失败原因'
);

CREATE INDEX idx_audit_time ON audit_log(operation_time DESC);
CREATE INDEX idx_audit_operator_time ON audit_log(operator, operation_time DESC);
CREATE INDEX idx_audit_type_time ON audit_log(operation_type, operation_time DESC);
CREATE INDEX idx_audit_target ON audit_log(target_type, target_id);
CREATE INDEX idx_audit_status_time ON audit_log(operation_status, operation_time DESC);
```

补充约束：

1. operation_time 存库用 DATETIME，接口返回时格式化为 yyyyMMddHHmm
2. operation_status 仅允许 0/1
3. error_msg 仅失败时写入
